## 1. Project Overview

This platform automates network configuration, validation, and monitoring in a
containerlab-based lab environment. The source of truth is **Nautobot**, and all
automation is built on **Nornir**, **pyATS/Genie**, and Python. Telemetry is
collected via **SNMP** (Prometheus SNMP Exporter) and **gNMI** (gnmic). The
final layers will add CI/CD (GitHub Actions) and an AI agent (NetClaw).

### 3.3 NAT

NAT overload on both Cisco routers:
- Inside interface: GigabitEthernet3
- Outside interface: GigabitEthernet2
- **IMPORTANT:** NAT_ACL must be an **extended** ACL to prevent inter-site
  double-NAT. The original standard ACL caused host-01 → host-02 ping failure.

**Correct NAT_ACL on rtr-01:**
```
ip access-list extended NAT_ACL
 deny   ip 192.168.1.0 0.0.0.255 10.0.0.0 0.255.255.255
 permit ip 192.168.1.0 0.0.0.255 any
 deny   ip 10.1.1.0 0.0.0.3 10.0.0.0 0.255.255.255
 permit ip 10.1.1.0 0.0.0.3 any
```

**Correct NAT_ACL on rtr-02:**
```
ip access-list extended NAT_ACL
 deny   ip 192.168.2.0 0.0.0.255 10.0.0.0 0.255.255.255
 permit ip 192.168.2.0 0.0.0.255 any
 deny   ip 10.2.2.0 0.0.0.3 10.0.0.0 0.255.255.255
 permit ip 10.2.2.0 0.0.0.3 any
```

> **Root cause of original ping failure:** rtr-02 was NATting return traffic
> (src: 192.168.2.10) to 10.0.0.2 with a standard ACL, so rtr-01 received
> packets with no matching translation entry and dropped them.

### 3.4 ACLs

- **arista-01:** `POLICY-HOST01-OUT` applied outbound on Ethernet2
  - `permit ip any 192.168.1.0/24`
  - `permit ip 192.168.1.0/24 any`

- **arista-02:** `POLICY-HOST02-OUT` applied outbound on Ethernet2
  - `permit ip 192.168.1.0/24 192.168.2.0/24`
  - `permit ip 10.0.0.0/30 192.168.2.0/24`

### 3.5 Host Routes

Set via `exec` commands in `containerlab/topology.yml`:
- host-01: `default via 192.168.1.1 dev eth1`
- host-02: `default via 192.168.2.1 dev eth1`

> **Note:** Arista switches load startup config from `*-initial.cfg`
> automatically. Cisco routers do **not** load startup configs on containerlab
> deploy — they must be reapplied manually from backup files or via automation.

### 5.3 gNMI / gnmic

**Why gNMI:** provides sub-second streaming telemetry vs SNMP's 30s polling,
and Arista cEOS has full OpenConfig model support.

**gnmic installed as host binary** (Docker images blocked by network policy):
```
/usr/local/bin/gnmic  version 0.45.0
```

**gnmic.yaml** (`telemetry/gnmic.yaml`):
```yaml
username: admin
password: admin
insecure: true
log: true
skip-verify: true

targets:
  172.20.20.13:6030:
    name: arista-01
    username: admin
    password: admin
    insecure: true
    skip-verify: true
  172.20.20.14:6030:
    name: arista-02
    username: admin
    password: admin
    insecure: true
    skip-verify: true

subscriptions:
  interfaces:
    paths:
      - /interfaces/interface/state/counters
      - /interfaces/interface/state/oper-status
    mode: stream
    stream-mode: sample
    sample-interval: 10s
    encoding: json_ietf

  bgp:
    paths:
      - /network-instances/network-instance[name=default]/protocols/protocol[name=BGP]/bgp/neighbors/neighbor/state
    mode: stream
    stream-mode: sample
    sample-interval: 30s
    encoding: json_ietf

  system:
    paths:
      - /system/state/hostname
      - /system/state/boot-time
    mode: stream
    stream-mode: sample
    sample-interval: 60s
    encoding: json_ietf

outputs:
  prometheus_output:
    type: prometheus
    listen: :9804
    path: /metrics
    metric-prefix: gnmic
    export-timestamps: true
    debug: false

target-subscriptions:
  172.20.20.13:6030:
    - interfaces
    - bgp
    - system
  172.20.20.14:6030:
    - interfaces
    - bgp
    - system
```

**docker-compose.yml** (`telemetry/docker-compose.yml`):

### Layer 9 — GitHub Actions CI/CD

Create `.github/workflows/netauto.yml` with three jobs:

1. **pre-check** — run `automation/validators/checks.py` pre-checks via Nornir
2. **deploy** — run `automation/tasks/deploy_config.py` to push configs
3. **post-check** — run validators and `automation/drift/detector.py` to confirm
   no drift after deploy

Trigger on: push to `main`, pull request to `main`.

The drift detector (`automation/drift/detector.py`) is the key CI gate — it
compares Nautobot-rendered desired state vs device running config using DeepDiff
and classifies severity as NONE / MINOR / MAJOR / CRITICAL. CI should fail on
MAJOR or CRITICAL.
